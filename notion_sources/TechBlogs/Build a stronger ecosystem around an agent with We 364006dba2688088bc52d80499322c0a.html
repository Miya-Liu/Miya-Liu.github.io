the practical way to choose between Skills, MCP, Plugins, Webhooks, Cron, direct API calls, and the Gateway/API Server

Short answer:
- Use a Skill when the integration is mostly “instructions + API examples”.
- Use MCP when you want to expose external capabilities as reusable tools, especially if an MCP server already exists.
- Use a Plugin when you want a native Hermes integration: custom tools, auth, providers, gateway/platforms, or deeper behavior.
- Use Webhooks when the outside world should trigger Hermes.
- Use Cron when Hermes should periodically check or pull something.
- Use direct terminal/curl/Python when it is a one-off or prototype.
- Use Gateway/API Server when another application wants to chat with or command Hermes as an agent service.

Direction matters.

External system → Hermes:

- Webhook
- Gateway API server
- Messaging gateway

Hermes → External system:

- Skill
- MCP
- Plugin
- Direct API/curl/Python calls

Time-based:

- Cron

Reusable agent knowledge:

- Skill

Reusable callable capability:

- MCP or Plugin

Durable multi-agent workflow:
- Kanban

A good practical rule:

  • Skill = “Hermes, here is how to do it.”
  • MCP = “Hermes, here are new tools you can call.”
  • Plugin = “Hermes, this is a native integration.”
  • Webhook = “External system, wake Hermes up when something happens.”
  • Cron = “Hermes, check this periodically.”
  • Gateway/API server = “External app, talk to Hermes as an agent service.”

  1. Skill: A Skill is a reusable instruction/runbook/prompt/API recipe. It tells the agent how to use existing tools: terminal, curl, Python, browser, file tools, etc.

Example:

“Use the internal analytics API. Auth is via ANALYTICS_API_KEY. Endpoint is /v1/query. Use this curl pattern. For errors, retry this way.”

Practical dependency constraints:

You need at least one of these to be true:

  1. The needed CLI/package is already installed in the agent environment.
  1. The skill includes install/setup instructions and the agent is allowed to install it.
  1. The capability is reachable through HTTP/curl and needs no local dependency.
  1. The dependency lives behind a remote API/service.
  1. The user manually prepared the environment before using the skill.

The dependency does not have to be open-source.

It can be:

  • public open-source package
  • private Python package
  • internal CLI
  • licensed commercial SDK
  • Dockerized tool
  • remote API
  • SaaS service
  • binary executable

But the agent must be able to access it.

Common Skill constraints:

  • No first-class tool schema.
  • Less reliable than MCP/plugin for repeated actions.
  • The model must read and follow instructions correctly.
  • Dependency installation may fail across OSes.
  • Harder to validate inputs/outputs.
  • Harder to hide secrets from the model.
  • Harder to enforce least privilege.
  • Not ideal for complex write operations.
  • Not ideal when many users/environments need identical behavior.
  • Not ideal when the integration requires OAuth or refresh tokens.
  • The skill itself cannot run code automatically; it only guides the agent.

When Skill is good:

  • Simple REST API
  • Internal runbook
  • Low-frequency workflow
  • Human-readable operational procedure
  • “Here is how to call this service”
  • Prototype before turning into MCP/plugin
  • Integration mostly involves curl/Python commands

When Skill is bad:

  • Safety-critical writes
  • Complex auth
  • Many endpoints
  • High reliability requirement
  • Need structured input/output validation
  • Need stable tool names
  • Need reusable tool calls across agents/apps
  • Need to hide credentials from the LLM
  • Need deployment across many machines

If your skill depends on some_private_sdk, then yes, that SDK must be available wherever the agent executes the skill’s commands. But it does not need to be open-source. It just needs to be installable/usable under your license and runtime constraints.

Even better: avoid local packages if a stable HTTP API exists. A curl-based skill is more portable than a Python-SDK-based skill.

  1. MCP: MCP exposes external capabilities as tools. Hermes connects to an MCP server, discovers its tools, and the agent calls them like native tools.

Example:

mcp_servers:
  company_api:
    url: "<https://mcp.company.com/mcp>"
    headers:
      Authorization: "Bearer ..."

or local:

mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "..."

Practical dependency constraints:

  • MCP has two main deployment modes.
    • Local stdio MCP server: Hermes launches a local process like npx ..., uvx ..., or your own binary.

      Constraints:

      • The command must be installed or installable.
      • Node/uv/Python may be required.
      • Server package must work on the agent’s OS.
      • Local credentials must be configured.
      • Hermes must restart to add/remove servers.
      • The MCP server process must be trusted enough to receive any env vars you pass.

      Hermes filters environment variables for stdio MCP servers. It does not pass all your API keys by default. You explicitly pass secrets in env.

    • Remote HTTP MCP server: Hermes connects to a remote server.

      Constraints:

      • Network access required.
      • Server must be available and authenticated.
      • You need TLS/auth/rate limits.
      • You operate or trust the remote MCP service.
      • Latency may be higher.
      • Tool behavior depends on remote server uptime/version.

MCP advantages:

  • First-class structured tools.
  • Clear schemas.
  • Reusable across multiple AI clients.
  • Better boundary than skills.
  • Can centralize auth and business logic.
  • Good for databases, GitHub, SaaS APIs, internal APIs.
  • Easier to make safe narrow tools like create_ticket instead of arbitrary curl.

MCP constraints:

  • Need an MCP server.
  • Local MCP may require Node/Python/uv/package install.
  • Remote MCP requires hosting.
  • Tool quality depends on server implementation.
  • Less Hermes-native than plugin.
  • Debugging crosses process/service boundary.
  • Trust boundary matters: untrusted MCP servers should receive minimal credentials.
  • Adding/removing servers usually needs restart.

When MCP is good:

  • You need reusable tools.
  • Other clients may also use the integration.
  • There is already an MCP server.
  • You want clean tool schemas.
  • You want to avoid modifying Hermes core.
  • You want database/API access with safer structured operations.

When MCP is bad:

  • You only need a simple one-off runbook.
  • You need very Hermes-specific behavior/UI/config.
  • You cannot host or run the MCP server reliably.
  • The dependency chain is too heavy for local setup.
  1. Plugin

What it is:

A Hermes plugin is a native extension. It can add tools, providers, memory backends, gateway platforms, auth logic, or deeper Hermes behavior.

Example:

~/.hermes/plugins/company_api/
  plugin.yaml
  __init__.py
  tools.py
  client.py

Practical dependency constraints:

  • Plugin code runs inside Hermes’s Python runtime.
  • Python dependencies must be installed in the Hermes environment or lazily installed.
  • Must be compatible with Hermes’s Python version and OS.
  • You need to maintain plugin code.
  • You need to design schemas, handlers, errors, auth, config.
  • If distributing, dependency pinning and supply-chain security matter.
  • Bad plugin code can affect the agent process directly.

Plugin advantages:

  • Most native Hermes integration.
  • First-class toolsets.
  • Can use Hermes config/profile/logging conventions.
  • Can conditionally expose tools based on credentials.
  • Better error handling.
  • Better auth integration.
  • Good for OAuth, token refresh, provider plugins, gateway/platform integrations.
  • Good for team-shared production integrations.

Plugin constraints:

  • More engineering work.
  • Tighter coupling to Hermes internals.
  • Needs tests and versioning.
  • Dependencies may bloat the environment if not handled carefully.
  • Higher blast radius than MCP because code runs in-process.
  • Must be maintained as Hermes evolves.

When Plugin is good:

  • You want a polished Hermes-native integration.
  • You need custom toolsets.
  • You need OAuth or credential management.
  • You need robust structured behavior.
  • You need custom gateway/platform/provider/memory functionality.
  • You are shipping the integration to a team or users.

When Plugin is bad:

  • You only need a simple API recipe.
  • You want the integration reusable outside Hermes.
  • You do not want to maintain Python plugin code.
  • You can use an existing MCP server instead.

  1. Webhook

What it is:

Webhook lets external systems trigger Hermes. The outside world sends POST requests to Hermes.

Example:

GitHub issue opened → POST to Hermes → Hermes triages issue → comments or sends Slack message.

Practical dependency constraints:

  • Hermes gateway must be running.
  • Webhook endpoint must be reachable from the external system.
  • Need HMAC secret/signature validation.
  • Need firewall/NAT/tunnel setup if local.
  • Payload template must match external event shape.
  • You must handle retries/idempotency for production.
  • Good event filtering is important to avoid spam/cost.
  • The external system must support webhooks or be wrapped by something that does.

Webhook advantages:

  • Best for push/event-driven automation.
  • No polling.
  • External service wakes Hermes up.
  • Can run full agent or deliver-only notification.
  • Good for GitHub, CI/CD, Stripe, monitoring, product events.

Webhook constraints:

  • It is a trigger, not a capability.
  • It does not by itself give Hermes access to query/update the service.
  • If Hermes needs to call the service after receiving event, you still need Skill/MCP/Plugin/API access.
  • Public endpoint security matters.
  • Payloads can be untrusted.
  • Need rate limiting and deduplication for noisy systems.

When Webhook is good:

  • “When X happens, ask Hermes to do Y.”
  • GitHub PR opened
  • CI failed
  • Stripe payment failed
  • Monitoring alert fired
  • User submitted form
  • Evaluation system detected performance drop

When Webhook is bad:

  • “Hermes should call this service on demand.”
  • “Hermes should query a database.”
  • “Hermes should manage Notion pages.”
  • Those require Skill/MCP/Plugin/direct API.
  1. Cron

What it is:

Cron schedules Hermes to run periodically.

Practical dependency constraints:

  • Hermes scheduler must be running.
  • The task prompt/script must be self-contained.
  • If it calls external services, credentials and dependencies must be available.
  • Good “silent if no changes” behavior is important.
  • For no-agent scripts, stdout is the notification body and empty stdout means no message.

Cron advantages:

  • Simple for polling and scheduled reports.
  • No external webhook support needed.
  • Good for health checks, daily summaries, scheduled evaluation.
  • Can pair with skills.

Cron constraints:

  • Polling can waste resources.
  • Not immediate unless interval is short.
  • Requires idempotency/watermarking.
  • Same integration dependencies as the underlying action.

When Cron is good:

  • “Every hour check evaluation metrics.”
  • “Every morning summarize failed jobs.”
  • “Every 30 minutes poll vendor incidents.”
  • “Nightly run agent benchmark.”

When Cron is bad:

  • External system can push events reliably.
  • You need real-time reaction.
  • You cannot afford repeated polling.
  1. Direct terminal/curl/Python

What it is:

Just ask Hermes to use terminal, curl, Python, browser, etc., without packaging anything as a skill/tool.

Practical dependency constraints:

  • Everything must already be installed or installable.
  • Commands must be safe enough to run.
  • Credentials must be available.
  • No durable reusable abstraction unless you later convert into a skill.

Advantages:

  • Fastest for prototypes.
  • No integration work.
  • Good for exploration.

Constraints:

  • Not repeatable.
  • Error-prone.
  • Harder to share.
  • No stable schemas.
  • Not a production integration.

Use it for:

  • First test
  • API exploration
  • One-time migration
  • Debugging
  • Generating examples for a future Skill/MCP/Plugin
  1. Gateway/API server

What it is:

External applications talk to Hermes as an agent service.

Practical dependency constraints:

  • Gateway/API server must be hosted and secured.
  • Need auth, rate limits, tenant isolation if used by apps.
  • Need session handling and delivery semantics.
  • If the agent needs external capabilities, still combine with Skill/MCP/Plugin.

Use it when:

  • Your app wants an “Ask Hermes” button.
  • A backend wants to submit tasks to Hermes.
  • Another system wants Hermes as an agent runtime.

Do not confuse with:

  • MCP/plugin/skill, which give Hermes capabilities.
  • Gateway/API server lets other systems use Hermes.

These approaches overlap. The practical distinction is mostly about direction, durability, reliability, and how much structure/safety you need.

Think of them as layers, not mutually exclusive choices:

Skill = knowledge/instructions
MCP = external tools
Plugin = native Hermes extension
Webhook = external event trigger
Cron = time trigger
Gateway = external app talks to Hermes
Direct API = prototype / one-off execution

In real systems, you often combine them.

Example:

“Evaluation score dropped, improve the agent.”

Could use:

  • Webhook: evaluator notifies Hermes that score dropped.
  • Skill: tells Hermes how to interpret eval reports and run curation.
  • MCP: exposes training/evaluation/deployment services as tools.
  • Plugin: provides native Hermes commands/toolsets for skill/model registry updates.
  • Cron: runs nightly benchmark if evaluator cannot push events.
  • Gateway/API server: lets dashboard or CI submit tasks to Hermes.

So the line is fuzzy.

The real question is not “which one is correct?” but:

“What role does this integration play?”

  1. Is it knowledge or capability?

If it is knowledge:

“Here is the procedure.”
“Here are the API endpoints.”
“Here is the runbook.”
“Here is how to debug this.”

Use Skill.

If it is capability:

“Call this function.”
“Query this database.”
“Create this ticket.”
“Deploy this model.”
“Search this private system.”

Use MCP or Plugin.

  1. Is it initiated by Hermes or by the outside world?

Hermes initiates:

  • Skill
  • MCP
  • Plugin
  • direct API/curl/Python

Outside world initiates:

  • Webhook
  • Gateway/API server

Time initiates:

  • Cron
  1. Is it occasional or repeated?

Occasional:

  • Direct API/curl/Python
  • Skill

Repeated:

  • Skill if simple
  • MCP if structured
  • Plugin if Hermes-native/production-grade
  1. Is it low-risk or high-risk?

Low-risk/read-only:

  • Skill is often enough.
  • MCP is better if repeated.

High-risk/write/deploy/delete/money/customer data:

  • Prefer MCP or Plugin.
  • Use narrow tools.
  • Add validation.
  • Add audit logs.
  • Add approval gates.
  • Avoid arbitrary shell/curl instructions.
  1. Is it local or remote?

Local dependency:

  • Skill can instruct Hermes to run local CLI/Python.
  • MCP can wrap local CLI/Python as a server.
  • Plugin can call local Python code directly.

Remote service:

  • Skill can call REST API via curl.
  • MCP can expose remote service tools.
  • Plugin can implement native client.
  • Webhook lets remote service trigger Hermes.

The same integration can evolve across approaches.

A common path:

Stage 0: Direct call

“Let me try curl against this API.”

Stage 1: Skill

“Now save the working procedure as a reusable skill.”

Stage 2: MCP

“This is used often; expose stable tools with schemas.”

Stage 3: Plugin

“This is important enough to be native Hermes functionality.”

Stage 4: Webhook/Cron

“Now automate when it runs.”

So it is not a decision tree with hard walls. It is a maturity ladder.

A useful mental model:

Skill vs MCP vs Plugin are mostly about “how Hermes gets capabilities.”

  • Skill: capability is implicit through instructions and existing tools.
  • MCP: capability is explicit through an external tool server.
  • Plugin: capability is explicit inside Hermes.

Webhook/Cron/Gateway are mostly about “who starts the work.”

  • Webhook: an external event starts Hermes.
  • Cron: time starts Hermes.
  • Gateway/API server: another app/user starts Hermes through an API or messaging surface.

Direct API/curl/Python is the scratchpad/prototype path.

Where the boundaries blur

  1. Skill can behave like a lightweight integration

If a skill says:

“Use curl to call this API with this schema.”

Then it effectively gives Hermes an integration, but not a formal tool.

The boundary with MCP/plugin blurs because the agent can still accomplish the same task.

Difference:

  • Skill relies on the model following instructions.
  • MCP/plugin gives structured callable tools.
  1. MCP can look like a plugin

If an MCP server exposes:

company_search_customer
company_update_ticket
company_create_invoice

then from the agent’s point of view it feels like a native toolset.

Difference:

  • MCP runs outside Hermes.
  • Plugin runs inside Hermes and can integrate more deeply with Hermes config/UI/auth.
  1. Plugin can wrap an MCP server

A plugin could manage config/auth and internally call an MCP server.

This is useful if you want native Hermes UX but still want a separate service boundary.

  1. Webhook can trigger a skill/MCP/plugin workflow

Webhook is not an alternative to Skill/MCP/Plugin. It often starts a workflow that uses them.

Example:

GitHub webhook arrives → Hermes loads github-code-review skill → calls GitHub MCP/plugin tools → posts review.

  1. Cron can replace webhook when push is unavailable

If a service cannot push events, Cron polls it.

Example:

  • Webhook version: “CI notifies Hermes when build fails.”
  • Cron version: “Hermes checks CI status every 10 minutes.”

Same workflow, different trigger.

The best dividing line

The closest thing to a hard line is this:

Trigger vs capability.

Triggers:

  • Webhook
  • Cron
  • Gateway/API server

Capabilities:

  • Skill
  • MCP
  • Plugin
  • direct API/curl/Python

Even that is not perfect, because a plugin can also add gateway/platform behavior, and a skill can describe scheduled workflows. But as a practical design rule, it is useful.

Second useful dividing line:

Instruction vs tool.

Instruction:

  • Skill

Tool:

  • MCP
  • Plugin

A skill tells the model what to do.
A tool gives the model something it can call.

Third useful dividing line:

External vs native.

External tool boundary:

  • MCP

Native Hermes extension:

  • Plugin

Human-readable procedure:

  • Skill

Practical examples

Example 1: Internal database

Prototype:

  • Direct Python or SQL client.

MVP:

  • Skill explaining read-only queries and constraints.

Production:

  • MCP server with safe query tools.

If deeply Hermes-native:

  • Plugin with company_db_query, company_db_lookup_customer, etc.

Trigger:

  • Cron checks daily anomaly report.
  • Webhook reacts when upstream system detects anomaly.

Example 2: Model training pipeline

Prototype:

  • Skill tells Hermes how to run scripts.

MVP:

  • Cron/Webhook triggers pipeline.
  • Skill documents curation/training/eval procedure.

Production:

  • MCP exposes:
    • list_eval_runs
    • curate_failures
    • start_training_job
    • evaluate_candidate
    • promote_model

Hermes-native production:

  • Plugin adds a training_pipeline toolset with auth, safety gates, model registry integration.

Example 3: Notion

Simple:

  • Skill using Notion REST API.

Repeated structured use:

  • MCP Notion server.

Native UX:

  • Plugin with OAuth and first-class Notion tools.

Event-driven:

  • Webhook from Notion/Zapier/Make triggers Hermes.

Example 4: Slack alert triage

Trigger:

  • Webhook from monitoring system.

Capability:

  • Skill for incident runbook.
  • MCP/plugin for log search.
  • MCP/plugin for ticket creation.
  • Slack gateway for delivery.

How I would choose in practice

Bottom line:

There is no hard division. The approaches are composable.

Use Skill to teach.
Use MCP to expose tools.
Use Plugin to make it native.
Use Webhook to react.
Use Cron to poll/schedule.
Use Gateway/API server to let apps talk to Hermes.
Use direct calls to prototype.
