Agent Ecosystem

Cloud, device, self-hosted, and hybrid agent systems.

20/04/2026 Miya Liu

Agent has diverged in many ways recently, especially in the realm of system or environment that host agents and how they interact with the physical world and human beings. Let's call the realm that hosts agents the agent ecosystem for now. Such realm has concored the cloud, devices (phones, laptops, etc.), and self-hosted servers, heading towards IoTs. It's a complex network and dynamic landscape, and I will try to decompose the components of the best of the design of it in this page. The analysis of such agent ecosystem is based on: Claude Code, Claude Cowork, Manus, OpenClaw, Cursor, and Korix. The goal is not to rank winners, but to kickoff a discussion where each system fits across cloud, device, and self-hosted workflows.

Scope note: product capabilities evolve fast. This survey captures positioning and typical usage patterns as of this draft.

Quick Comparison

Platform Primary Mode Best Fit Trade-off
Claude Code Agentic coding assistant (CLI / IDE) Codebase understanding, implementation, debugging, review loops Depends on model/tool access and engineering context quality
Claude Cowork Desktop knowledge-work agent Delegating recurring or document-heavy office workflows Task quality depends on clear instructions and guardrails
Manus Cloud execution agent Autonomous, multi-step tasks that produce finished deliverables Can consume credits quickly on long/error-prone workflows
Cursor AI-first code editor + agent Developer-in-the-loop coding with strong IDE ergonomics Still requires workflow discipline for large-scale changes
OpenClaw Extensible/self-host-friendly framework Custom agent pipelines, plugin architecture, channel integrations Higher setup and operations effort than managed products
Korix Domain/enterprise AI implementation tracks Sector-specific or enterprise process automation engagements Positioning varies by Korix product line and deployment model

Interaction Playground

What it feels like: you propose an edit plan, then iterate through short cycles (diff → verify → refine).

When it shines: code edits, test/debug loops, and “make this change without breaking X”.

Tool surface: terminal commands, file operations, and repository-aware search/edits.

Practical habit: explicitly name the “expected outputs” so tool runs stay grounded.

Artifact shape: patches, code diffs, and verification logs.

Review style: treat each checkpoint as a reviewable unit—not a single monolithic request.

Memory flavor: project context + working notes (what you’ve built so far) is repeatedly re-used.

Design idea: keep “canonical decisions” in files so the agent can anchor on them.

Governance pattern: you approve the next tool step (implicitly via iteration cadence, or explicitly via config).

Failure mode: unclear acceptance criteria makes the agent brute-force search paths.

What it feels like: the IDE becomes your control panel—small steps, fast feedback, and frequent “stop & inspect”.

When it shines: refactors, multi-file edits, and “maintain style/structure”.

Tool surface: file system awareness + IDE commands (run, inspect, re-run).

Practical habit: ask for explicit checkpoints (“show me the diff before applying”).

Artifact shape: suggested edits/diffs at the place you’re working.

Review style: accept as you go; reject/redirect when constraints drift.

Memory flavor: current file context + recent conversation anchors.

Design idea: keep “invariant rules” as project notes (so the agent won’t re-invent them).

Governance pattern: your attention is the gate—IDE checkpoints help you notice risky changes early.

Failure mode: long requests without intermediate diff checkpoints can create cascading edits.

What it feels like: you delegate a workflow, then review drafts/outputs at milestones.

When it shines: document-heavy tasks (summaries, reports, recurring ops).

Tool surface: file access, app-level actions, and structured task steps.

Practical habit: name the “inputs” (folders/docs) and “deliverables” (templates/pages).

Artifact shape: drafts, stitched notes, and ready-to-send documents.

Review style: focus on tone/consistency + correctness before sign-off.

Memory flavor: “working set” files and session context.

Design idea: keep your templates and style guide somewhere the agent can reference.

Governance pattern: explicit review checkpoints reduce the risk of wrong assumptions propagating.

Failure mode: vague deliverable format creates rework cycles.

What it feels like: you “checkpoint” by iterating prompts and asking for specific intermediate outputs.

When it shines: fast brainstorming, spec drafts, and narrowing requirements.

Tool surface: browsing/code-interpreter style tools, or API tool integrations.

Practical habit: request tool outputs in a structured form (steps, assumptions, artifacts).

Artifact shape: text drafts, outlines, and “ready-to-edit” sections you can apply elsewhere.

Review style: verify claims, then convert drafts into your target format (tickets, patches, notes).

Memory flavor: what you provide in the prompt plus any product-level memory features.

Design idea: keep your “current truth” as a short, explicit summary the agent must reuse.

Governance pattern: system prompts, refusal criteria, and user-specified constraints.

Failure mode: ambiguous constraints can lead to confident but mismatched outputs.

What it feels like: you scope the mission, then check progress at stage boundaries.

When it shines: multi-step workflows that end in a completed artifact.

Tool surface: cloud execution steps with consistent state handling.

Practical habit: define “done” as measurable outputs.

Artifact shape: bundled results (files, summaries, staged deliverables).

Review style: treat evaluation as part of the pipeline: verify, then iterate.

Memory flavor: the workflow state lives inside the run (plus any persistent context you provide).

Design idea: log decisions so future runs don’t restart from scratch.

Governance pattern: operational constraints (credits/budgets, run limits) shape how you should prompt.

Failure mode: long ambiguous missions can become expensive exploration.

What it feels like: each workflow step is a controllable boundary (transform → verify → commit).

When it shines: custom pipelines and “agent as a component”.

Tool surface: plugins + integrations that define what the agent can touch.

Practical habit: build explicit tool contracts (inputs/outputs) so reasoning stays stable.

Artifact shape: pipeline outputs and intermediate traces.

Review style: evaluate step-by-step, then promote only verified results.

Memory flavor: typically more customizable—session/project/global stores you can configure.

Design idea: separate “working memory” from “promoted knowledge”.

Governance pattern: you encode policy gates in the orchestration layer.

Failure mode: missing validation steps lets untrusted outputs progress too far.

What it feels like: progress is tied to compliance checkpoints and approved steps.

When it shines: process-heavy and regulated environments.

Tool surface: integrations aligned to enterprise workflows.

Practical habit: define required approvals for sensitive operations.

Artifact shape: structured reports and outputs with traceability.

Review style: verify provenance and compliance attributes.

Memory flavor: organization context and curated data sources.

Design idea: ensure the agent has “approved knowledge” boundaries.

Governance pattern: approvals + policy gates drive the interaction flow.

Failure mode: policy drift makes outputs hard to trust.

    Platform Notes
    
          
      
            
    • Claude Code: strongest when tasks involve code edits, shell tools, iterative verification, and repository-aware reasoning.
      • 
            
    • Claude Cowork: targets knowledge-work delegation on desktop workflows (files, recurring tasks, synthesis).
      • 
            
    • Manus: emphasizes autonomous execution and end deliverables, often useful for research-to-output pipelines.
      • 
            
    • Cursor: pairs a capable coding agent with IDE-native developer experience and checkpoint-style iteration.
      • 
            
    • OpenClaw: attractive for teams that need control, extensibility, and custom orchestration over channels/tools.
      • 
            
    • Korix: appears in more than one market context; validate the exact Korix product before architecture decisions.
      • 
          
    

          
    Cloud vs Device vs Self-Hosted
    
          
      
            
    • Cloud-first agents are great for elastic execution and cross-system automation, but need strong governance for data and spend.
      • 
            
    • Device-side agents improve latency and local context handling, but are bounded by endpoint compute and environment variance.
      • 
            
    • Self-hosted ecosystems maximize control and compliance, but shift complexity to infra, reliability, and maintenance.
      • 
            
    • Hybrid architecture is often the practical default: local context + cloud reasoning + policy-based routing of sensitive tasks.
      • 
          
    

          
    Suggested Selection Heuristic
    
          
      
            
    1. If your core workflow is software delivery, start with Claude Code or Cursor.
      2. 
            
    2. If your core workflow is business/knowledge operations, evaluate Claude Cowork and Manus.
      3. 
            
    3. If you need deep customization or channel-level control, prototype with OpenClaw.
      4. 
            
    4. If you are in a regulated or process-heavy enterprise environment, evaluate Korix-style implementation models with clear governance criteria.
