Agent Self-Evolution

How agents learn, adapt, and improve over time.

20/04/2026 Miya Liu

This page is my ongoing study log about agent self-evolution. I will track how agents improve from interaction history, how memories and skills are updated, and how to keep the evolution loop reliable and safe in production.

What Self-Evolution Means

  • Not one-shot fine-tuning, but iterative adaptation in running systems.
  • Signals come from failures, corrections, confirmations, and repeated tasks.
  • Outputs include updated memory, adjusted prompts, and extracted skills.

Core Loop to Study

  1. Observe outcomes (successes, failures, friction points).
  2. Extract structured learning from transcripts and tool traces.
  3. Classify learning (memory update, rule update, skill candidate).
  4. Validate changes with tests or replay tasks.
  5. Promote reliable learnings and monitor drift.

Memory and Skill Evolution

  • Short-term memory supports immediate context adaptation.
  • Project/team memory captures durable conventions and known pitfalls.
  • Reusable skills are distilled from repeated, verified learnings.

Evaluation Dimensions

  • Task success and recovery quality after errors.
  • Human supervision load per completed task.
  • Stability over time (avoiding regressions and behavior drift).
  • Cost-performance balance under realistic workloads.

Guardrails

  • Do not auto-promote uncertain learnings into global rules.
  • Keep traceability from behavior change to source evidence.
  • Use staged rollout for major policy or prompt updates.

Evolution Metrics Framework (v1)

This section defines the primary evolution metrics dimensions used by the verification layer. Terminology: each metric must be backed by explicit computation rules in scorecard implementation.

1) Skill Evolution

Purpose: evaluate whether capability growth is real, transferable, and non-destructive.

Metrics:

  • skill_expansion: additive feature unlocked. Measures success on newly introduced capability slices relative to baseline.
  • skill_enhancement: improved performance of existing skills on OOD or harder variants. Measures delta on fixed “same skill, harder context” slices.
  • skill_intactness: previously consolidated skills remain intact during expansion/enhancement. Measures regression rate on stable retention slices (catastrophic forgetting indicator).

Suggested benchmark linkage:

  • Primary: EvoSkill, WildClawBench.
  • Supporting: Protocol B seen/unseen family splits for transfer vs memorization.

2) Memory Adaptation

Purpose: evaluate whether memory is collected, refreshed, retrieved, and useful under realistic multi-session operation.

Core metrics:

  • remembered: memory system captures useful cross-session and long-session information.
  • renewed: memory stays up to date and tracks migration (external source-of-truth changes and user preference drift).
  • recalled: relevant memory is retrieved when current agent state requires it.
  • consistent: no obvious conflicts or near-duplicate memory artifacts.
  • valuable: memory contributes to robustness or efficiency improvement.

Suggested additional memory aspects (recommended but on-hold):

  • provenance: each memory item links to source event/session and timestamp.
  • staleness_control: stale/conflicting memories are decayed, superseded, or invalidated.
  • retrieval_precision: retrieved memories are relevant, not over-retrieved.
  • retrieval_latency: memory lookup overhead stays within budget.
  • privacy_safety: restricted or sensitive memory is not inappropriately surfaced.

Suggested benchmark linkage:

  • Primary: MemoryArena.
  • Supporting: Protocol C feedback-event lineage and update-boundary auditing.

3) Robustness Evolution

Purpose: evaluate whether the agent remains reliable under failures, perturbations, and adversarial context.

Metrics:

  • self_correction: detect and recover from execution errors and environment perturbations.
  • noise_rejection: reject unsafe/misleading injections and poison context, while ignoring irrelevant noise.
  • evolution_stability: newly evolved behaviors do not degrade ongoing execution quality, including stability under continual memory growth (recall quality should not collapse as memory expands).

Suggested benchmark linkage:

  • Primary: SEA-Eval (sequential stability), perturbation suites in Protocol A/B/C.
  • Supporting: operator-based perturbation methodology similar to AutoEvoEval.

4) Efficiency Convergence

Purpose: evaluate whether evolution reduces human and compute cost while maintaining quality.

Metrics:

  • decreased_user_intervention: fewer user clarifications/corrections to reach acceptable outcomes.
  • effective_steps_reduction: fewer actions/tool calls/turns to completion on comparable tasks.
  • token_consumption_reduction: lower or non-increasing token cost on comparable tasks.
  • efficiency_evolution_rate: success rate normalized by token cost.
    v1 formula: success_rate / avg_tokens

Measurement policy:

  • For comparable tasks, use non-increase or statistically significant decrease according to configured threshold policy.
  • Exact thresholds MUST be persisted in eval_config_canonical_json.

Suggested benchmark linkage:

  • Primary: SEA-Eval token trajectory (T) and sequential SR/T analysis.
  • Supporting: existing suite-level analysis (to be filled).
