Agents: From Design to Infrastructure

Bridging product intent, system architecture, and production ops.

21/04/2026 Miya Liu

This page is a working outline for how I think about agents as a full stack: what we design for users and operators, how the agent system is structured, and what infrastructure must exist to run it reliably. I will expand each section with examples and trade-offs as the study deepens.

Design

  • User goals, trust boundaries, and failure modes the product must handle gracefully.
  • Task decomposition: single-turn vs multi-step workflows and human-in-the-loop points.
  • Transparency: what the agent is allowed to do, and how users can audit or undo actions.

System architecture

  • Model routing, context assembly, and tool/MCP boundaries.
  • Memory tiers: session, project, and durable stores; when to read vs write.
  • Orchestration: one agent vs sub-agents, queues, and idempotent side effects.

Infrastructure

  • Runtime: where agents execute (cloud, device, self-hosted) and latency budgets.
  • Scaling: concurrency, rate limits, and cost controls.
  • Observability: logs, traces, and evaluation of task success over time.
  • Security and compliance: secrets, data residency, and least-privilege tool access.

Connecting the layers

  • Design choices should map to explicit architecture contracts (e.g. which tools exist).
  • Architecture choices should map to infra SLOs (availability, recovery, rollback).
  • Infra feedback should inform design (what is feasible to supervise at scale).
