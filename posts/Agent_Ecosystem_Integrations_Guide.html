Build a stronger ecosystem around an agent

— webhooks, MCP, plugins, skills, cron, and gateway (Hermes-oriented notes).

18/05/2026 Miya Liu

Notes on the practical way to choose between skills, MCP, plugins, webhooks, cron, direct API calls, and a gateway/API server when you extend an agent (examples use the Hermes stack naming from the original draft).

Short answer

  • Use a skill when the integration is mostly instructions plus API examples.
  • Use MCP when you want external capabilities as reusable tools, especially if an MCP server already exists.
  • Use a plugin when you want a native integration: custom tools, auth, providers, gateway or platforms, or deeper behavior.
  • Use webhooks when the outside world should trigger the agent.
  • Use cron when the agent should periodically check or pull something.
  • Use direct terminal/curl/Python when it is a one-off or prototype.
  • Use a gateway/API server when another application wants to chat with or command the agent as a service.

Direction matters

External system → agent: webhook, gateway API server, messaging gateway.

Agent → external system: skill, MCP, plugin, direct API/curl/Python calls.

Time-based: cron.

Reusable agent knowledge: skill.

Reusable callable capability: MCP or plugin.

Durable multi-agent workflow: Kanban (and similar coordination surfaces).

Worked examples

The subsections below expand each option with criteria, Hermes-oriented setup hints, and upgrade paths.

Skill approach

Use a Skill when the integration is mostly operational knowledge.

A skill is not a live connector by itself. It tells Agent how to use something using existing tools like terminal, curl, file editing, browser, Python scripts, etc.

Good for

  • REST APIs that are simple
  • Internal runbooks
  • “How to use this service”
  • One-off or low-frequency integrations
  • API recipes
  • Human conventions
  • Troubleshooting steps
  • Prompting style for an integration

Bad for:

  • Complex auth flows
  • Many repetitive calls
  • Safety-critical write operations
  • Large schemas
  • Returning structured tool results
  • Hiding credentials from the model
  • Things you want exposed as first-class tools

Example: Notion as a Skill

Hermes can use a Notion skill that says how to call the Notion API via curl with `NOTION_API_KEY`.

Setup:

1. Create a Notion integration:

https://notion.so/my-integrations

2. Put the key in:

~/.hermes/.env

Example:

NOTION_API_KEY=***

3. Share the target Notion page/database with the integration in Notion:

Page menu → Connect to → your integration.

4. Load the skill:

In chat:

/skill productivity/notion

Or start Hermes with:

hermes -s productivity/notion

Then ask:

“Search my Notion workspace for pages about Q4 planning.”

or:

“Create a new Notion page in database X with title Y and these notes.”

When to upgrade from Skill to MCP/Plugin:

If you start asking the same operations repeatedly, for example:

  • create task
  • update status
  • query database
  • append meeting notes
  • sync pages
  • handle pagination
  • validate database properties

Then skill-only becomes fragile. Move to MCP or plugin.

MCP approach

Use MCP when you want Agent to gain external tools from an MCP server.

MCP is best when:

  • An MCP server already exists for your target system.
  • You want the same server usable by Hermes, Claude Desktop, Cursor, other agents, etc.
  • You want a clean tool boundary.
  • You want a remote service to expose tools over HTTP.
  • You want to avoid modifying Hermes code.
  • You want tools auto-discovered at startup.

Good examples:

  • Filesystem MCP
  • GitHub MCP
  • Postgres MCP
  • Notion MCP
  • Slack MCP
  • Browser/data API MCP
  • Internal company MCP server

Example stdio MCP server config:

mcp_servers:
  time:
    command: "uvx"
    args: ["mcp-server-time"]

Example Node/npm MCP server:

mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "ghp_xxxxxxxxx"

Example remote HTTP MCP server:

mcp_servers:
  company_api:
    url: "https://mcp.mycompany.com/v1/mcp"
    headers:
      Authorization: "Bearer sk_xxxxxxxxx"
      X-Team-Id: "engineering"
    timeout: 180
    connect_timeout: 30

Then restart Hermes.

Check:

hermes mcp list
hermes mcp test company_api

Tool naming:

If server name is `company_api` and tool name is `search_customers`, Hermes exposes something like:

mcp_company_api_search_customers

MCP is especially good for remote databases.

Example: remote Postgres database

Recommended pattern:

mcp_servers:
  postgres_ro:
    command: "npx"
    args: ["-y", "some-postgres-mcp-server"]
    env:
      DATABASE_URL: "postgresql://readonly_user:***@host:5432/db"

Important deployment advice:

  • Use a read-only database user by default.
  • Create separate tools for writes; do not give arbitrary SQL write access casually.
  • Put secrets in env/config, not in prompts.
  • Prefer allowlisted queries or database views for production.
  • Use timeouts and row limits.
  • Log writes.

Use MCP if:

“Hermes should query this database as part of many tasks.”

Use webhook/cron instead if:

“The database should notify Hermes when something changes.”

Plugin approach

Use a Hermes Plugin when you want a native Hermes integration.

A plugin is more work than a skill or MCP config, but it is the most Hermes-native option.

Use a plugin when:

  • You need custom Hermes tools.
  • You want a toolset like `spotify`, `notion`, `company_api`.
  • You need custom auth handling.
  • You need OAuth support.
  • You need structured errors/results.
  • You want tools to appear/disappear based on credentials.
  • You want to ship the integration to other Hermes users.
  • You need gateway/platform/provider/memory/context behavior, not just tool calls.
  • You want deep integration with Hermes config, auth, logs, and UI.

Example plugin layout:

~/.hermes/plugins/company_api/
  plugin.yaml
  __init__.py
  tools.py
  client.py

Example plugin.yaml:

name: company_api
version: 1.0.0
description: "Company API integration for customers, invoices, and alerts."
author: Your Name
kind: backend
provides_tools:
  - company_api_customers
  - company_api_invoices
  - company_api_alerts

Example tools.py pattern (abbreviated structure):

import os
from tools.registry import registry, tool_result, tool_error

def _check_company_api_available():
    return bool(os.getenv("COMPANY_API_KEY"))

def _handle_company_customers(args, **kw):
    action = args.get("action", "search")
    try:
        if action == "search":
            query = args.get("query", "")
            # call your API here
            return tool_result({
                "success": True,
                "action": action,
                "results": []
            })
        return tool_error(f"Unknown action: {action}")
    except Exception as exc:
        return tool_error(
            f"company_api_customers failed: {type(exc).__name__}: {exc}"
        )

registry.register(
    name="company_api_customers",
    toolset="company_api",
    schema={
        "name": "company_api_customers",
        "description": "Search and retrieve company customer records.",
        "parameters": {
            "type": "object",
            "properties": {
                "action": {
                    "type": "string",
                    "enum": ["search", "get"],
                    "default": "search"
                },
                "query": {
                    "type": "string",
                    "description": "Search query"
                },
                "customer_id": {
                    "type": "string",
                    "description": "Customer ID for get action"
                }
            }
        }
    },
    handler=_handle_company_customers,
    check_fn=_check_company_api_available,
    requires_env=["COMPANY_API_KEY"],
)

Then:

1. Put secret in ~/.hermes/.env:

COMPANY_API_KEY=***

2. Restart Hermes.

3. Enable/check tools:

hermes tools list
hermes tools enable company_api

Use plugin instead of MCP when:

  • The integration is Hermes-specific.
  • You need tight Hermes auth/profile/toolset behavior.
  • You want first-class UX.
  • You are contributing it to Hermes.
  • You need custom gateway behavior or model provider behavior.

Use MCP instead of plugin when:

  • You want the integration to be agent-framework-agnostic.
  • A good MCP server already exists.
  • You want remote/shared tools.
  • You do not want to write Hermes-specific code.

Webhook approach

Use Webhook when the external system initiates the interaction.

This is the right model for:

  • GitHub event → Hermes reviews/triages
  • Stripe event → Hermes summarizes payment issue
  • CI failure → Hermes investigates
  • Monitoring alert → Hermes suggests remediation
  • IoT event → Hermes reacts
  • External app button → POST task to Hermes

Setup:

First enable webhook platform.

Option A:

hermes gateway setup

Option B: edit ~/.hermes/config.yaml:

platforms:
  webhook:
    enabled: true
    extra:
      host: "0.0.0.0"
      port: 8644
      secret: "generate-a-strong-secret-here"

Start gateway:

hermes gateway run

or service:

hermes gateway install
hermes gateway start

Verify:

curl http://localhost:8644/health

Create subscription (GitHub issues example):

hermes webhook subscribe github-issues \
  --events "issues" \
  --prompt "New GitHub issue #{issue.number}: {issue.title}

Action: {action}
Author: {issue.user.login}
Body:
{issue.body}

Please triage this issue." \
  --deliver telegram \
  --deliver-chat-id "-100123456789"

For a remote API service:

hermes webhook subscribe vendor-alerts \
  --prompt "Vendor API alert: {alert.name}
Severity: {alert.severity}
Message: {alert.message}

Analyze the likely cause and suggest next steps." \
  --deliver discord \
  --deliver-chat-id "1234567890"

For no-LLM direct delivery:

hermes webhook subscribe deploy-events \
  --deliver telegram \
  --deliver-chat-id "123456789" \
  --deliver-only \
  --prompt "Deployment {service} version {version}: {status}"

Use webhook if:

“When X happens outside Hermes, run Hermes.”

Do not use webhook if:

“Hermes needs to call X on demand.”

For that, use Skill/MCP/Plugin.

Cron approach

Use Cron when there is no external event, but you want Hermes to check periodically.

Good for:

  • Polling APIs
  • Daily summaries
  • Monitoring dashboards
  • Database checks
  • Report generation
  • “Every hour check for failed jobs”
  • “Every morning summarize Notion tasks”

Example: daily Notion task summary

hermes cron create "0 9 * * *"

Prompt:

Use the productivity/notion skill. Query my Notion Tasks database for items where Status is not Done and Due Date is today. Summarize the top priorities and deliver a concise morning plan.

If using the cron tool from chat, equivalent fields would be:

schedule: "0 9 * * *"
skills: ["productivity/notion"]
prompt: "Query my Notion Tasks database..."

Example: remote API polling

“Every 30 minutes, call the vendor status API. If there are incidents affecting our services, summarize impact and recommended action. If nothing changed, stay silent.”

For pure script watchdogs, use no-agent mode:

  • Script prints a message only when alert condition is met.
  • Empty stdout means no notification.
  • No LLM cost.

Use cron if:

“Hermes should check on a schedule.”

Use webhook if:

“The service can push events to Hermes.”

Direct API, curl, or Python

Use this for prototypes and one-off tasks.

Example:

“Call this API once and summarize the result.”

Hermes can use terminal/curl/Python directly if credentials are available. This is fastest, but least productized.

Good for:

  • Testing API access
  • Exploring payloads
  • One-time migration
  • Debugging
  • Building a future skill/plugin/MCP server

Bad for:

  • Repeated production use
  • Complex auth
  • Safety-critical writes
  • Sharing with others
  • Stable UX

Upgrade path:

direct curl → skill → MCP/plugin

Gateway and API server

Use this when an outside application wants to talk to Hermes as an agent.

This is different from “integrating a service as a tool”.

Examples:

  • Your web app has an “Ask Hermes” button.
  • Your internal dashboard sends support tickets to Hermes.
  • Another backend service asks Hermes to analyze logs.
  • A mobile app uses Hermes as its agent backend.

Options:

  • Webhook: simple one-way event/task submission.
  • API Server gateway: more application-like API integration.
  • Messaging gateway: Telegram/Discord/Slack/etc. as the user-facing surface.

Use webhook when:

“POST event, get processing/delivery.”

Use API server when:

“My app wants a general agent API.”
